# Builder Prompt: PubSub Phase P3-P4 — Back-Channeling + Topic Routing

## Objective

Implement Phase P3 (Back-Channeling) and Phase P4 (Topic Routing & Security) of
the AAFP PubSub track. P3 adds a side-channel for progress/lifecycle events
during long-running RPCs, delivered via PubSub subscriptions with no wire
protocol changes. P4 adds MQTT-style hierarchical topics, `+`/`#` wildcard
matching, a `TopicMatcher` implementation, reserved topic prefixes, and the
unified view of streaming RPC as specialized PubSub.

## Prerequisites

- P1 (Local PubSub Surface) and P2 (Networked Propagation) must be complete —
  `NetworkedPubSub` is wired into `ServingAgent`, `subscribe()`/`publish()`
  work end-to-end across peers, and the propagation driver forwards floodsub
  publishes.
- P2.8 (Server-Streaming RPC + Cancellation) must be complete — back-channeling
  reuses `CancellationToken` and the bi-stream request loop.
- RFC-0006 extension machinery must exist in `aafp-messaging/src/extensions.rs`
  (extension encode/decode on `RPC_REQUEST` frames).

## What to Build

### Part 1: Back-Channel Topic Naming (P3, §5.2)

Back-channel topics follow the convention:

```
rpc.<server_agent_id>.<request_id>.progress
```

- `<server_agent_id>` — the AgentId of the server handling the RPC.
- `<request_id>` — a unique, unguessable request id (64+ random bits, hex or
  base32url encoded) generated by the client per call. Unguessability is a
  security requirement (§11.3): a malicious peer must not be able to predict
  the topic and inject spoofed progress events.

Add a helper that constructs and validates back-channel topics:

```rust
/// Construct a back-channel topic for an in-flight RPC.
///
/// Format: `rpc.<server_id>.<req_id>.progress`
pub fn backchannel_topic(server_id: &AgentId, req_id: &str) -> String {
    format!("rpc.{}.{}.progress", server_id, req_id)
}

/// Validate that a topic is a well-formed back-channel topic.
pub fn is_backchannel_topic(topic: &str) -> bool {
    let segs: Vec<&str> = topic.split('.').collect();
    segs.len() == 4
        && segs[0] == "rpc"
        && segs[3] == "progress"
        && !segs[1].is_empty()
        && !segs[2].is_empty()
}

/// Extract the (server_id, request_id) from a back-channel topic.
/// Returns `None` if the topic is malformed.
pub fn parse_backchannel_topic(topic: &str) -> Option<(String, String)> {
    let segs: Vec<&str> = topic.split('.').collect();
    if segs.len() == 4 && segs[0] == "rpc" && segs[3] == "progress" {
        Some((segs[1].to_string(), segs[2].to_string()))
    } else {
        None
    }
}
```

The `req_id` generator:

```rust
use rand::Rng;

/// Generate an unguessable 128-bit request id, base32url-encoded (26 chars).

    let bytes: [u8; 16] = rand::thread_rng().gen();
    base32::encode(base32::Alphabet::Crockford, &bytes)
}
```

### Part 2: RFC-0006 Extension to Advertise Back-Channel Topic (P3, §5.5)

Register a new extension type that carries the back-channel topic on the
`RPC_REQUEST` frame. Servers that do not understand the extension ignore it
(RFC-0006 graceful degradation), so the call degrades to a plain unary RPC.

```rust
// aafp-messaging/src/extensions.rs

/// Extension type 0x0010: back-channel topic.
///
/// Payload: CBOR-encoded text string (the back-channel topic name).
/// Carried on RPC_REQUEST frames. Servers that do not recognize this
/// extension MUST ignore it (RFC-0006 §3 graceful degradation), causing
/// the RPC to proceed as a plain unary call with no progress events.
pub const EXT_BACKCHANNEL_TOPIC: u16 = 0x0010;

/// Encode a back-channel topic as an RFC-0006 extension payload (CBOR text).
pub fn encode_backchannel_ext(topic: &str) -> Result<Vec<u8>, CborError> {
    let mut buf = Vec::new();
    ciborium::into_writer(&topic, &mut buf)?;
    Ok(buf)
}

/// Decode a back-channel topic from an extension payload.
/// Returns `None` if the payload is not a valid CBOR text string.
pub fn decode_backchannel_ext(payload: &[u8]) -> Option<String> {
    ciborium::from_reader::<String, _>(payload).ok()
}

/// Extract the back-channel topic from a frame's extension list, if present.
pub fn extract_backchannel_topic(frame: &Frame) -> Option<String> {
    frame
        .extensions()
        .iter()
        .find(|e| e.typ == EXT_BACKCHANNEL_TOPIC)
        .and_then(|e| decode_backchannel_ext(&e.data))
}
```

Wire it into the RPC request builder:

```rust
/// Attach a back-channel topic extension to an RPC request frame.
pub fn frame_with_backchannel(mut frame: Frame, bc_topic: &str) -> Frame {
    match encode_backchannel_ext(bc_topic) {
        Ok(data) => {
            let ext = Extension { typ: EXT_BACKCHANNEL_TOPIC, data };
            frame.add_extension(ext);
        }
        Err(e) => {
            tracing::warn!("failed to encode backchannel extension: {e}");
            // Proceed without extension — degrades to unary RPC.
        }
    }
    frame
}
```

### Part 3: Backward Compatibility — Graceful Degradation (P3, §5.5)

The back-channel is **opt-in and backward compatible**. The degradation rules:

1. **Server does not understand `EXT_BACKCHANNEL_TOPIC`**: the extension is
   ignored per RFC-0006. The server processes the RPC as a normal unary call.
   The client's progress stream simply never receives events (it closes when
   the RPC response arrives). No error, no hang.

2. **Server understands the extension but has no `backchannel_handler`**: the
   server falls back to the regular `handler()` if one is registered. The
   back-channel topic is known but never published to. Same client behavior as
   case 1.

3. **Client does not request a back-channel**: no extension is attached. The
   server processes the RPC normally. Existing unary RPC code paths are
   untouched.

Implement this in the server request loop:

```rust
// Inside ServingAgent request loop, after decoding the RPC_REQUEST frame:
let bc_topic = extract_backchannel_topic(&frame);

let handler_result = if let Some(ref topic) = bc_topic {
    if let Some(ref bc_handler) = self.backchannel_handler {
        // Back-channel path: create Backchannel, run handler.
        let bc = Backchannel::new(topic.clone(), Arc::clone(&self.pubsub), self.our_id);
        let cancel = CancellationToken::new();
        let cancel_monitor = monitor_stream_close(&recv, cancel.clone());
        tokio::select! {
            r = bc_handler(req.clone(), bc, cancel.clone()) => r,
            _ = cancel_monitor => Err("client disconnected".into()),
        }
    } else if let Some(ref handler) = self.handler {
        // Extension present but no backchannel handler — degrade to unary.
        handler(req.clone()).await
    } else {
        Err("no handler registered".into())
    }
} else {
    // No extension — plain unary RPC (existing path, unchanged).
    if let Some(ref handler) = self.handler {
        handler(req.clone()).await
    } else {
        Err("no handler registered".into())
    }
};
```

### Part 4: `Backchannel` Type and Server Handler (P3, §5.2-5.3)

The `Backchannel` handle is given to a handler so it can emit progress events
to the back-channel topic via the local `NetworkedPubSub` instance.

```rust
// aafp-sdk/src/simple.rs

/// Handle given to a handler so it can emit back-channel events during a
/// long-running RPC. Events are published to the back-channel topic via
/// the local PubSub instance and forwarded to subscribers (the client)
/// by the propagation driver.
#[derive(Clone)]
pub struct Backchannel {
    topic: String,
    pubsub: Arc<NetworkedPubSub>,
    our_id: AgentId,
}

impl Backchannel {
    pub(crate) fn new(topic: String, pubsub: Arc<NetworkedPubSub>, our_id: AgentId) -> Self {
        Self { topic, pubsub, our_id }
    }

    /// The back-channel topic name (`rpc.<server>.<req_id>.progress`).
    pub fn topic(&self) -> &str {
        &self.topic
    }

    /// Emit a progress event with a percentage and message.
    pub async fn progress(&self, percent: u8, msg: impl Into<String>) {
        let body = format!("[{}%] {}", percent, msg.into());
        let event = Event::text(body);
        let _ = self.pubsub.publish_local(&self.topic, self.our_id, event.encode());
    }

    /// Emit a structured partial result (raw bytes).
    pub async fn partial(&self, data: Vec<u8>) {
        let _ = self.pubsub.publish_local(&self.topic, self.our_id, data);
    }

    /// Emit a structured partial result as a CBOR map (recommended for
    /// typed payloads, per §4.2 AutoGen lesson).
    pub async fn partial_cbor(&self, map: &impl serde::Serialize) {
        let mut buf = Vec::new();
        if ciborium::into_writer(map, &mut buf).is_ok() {
            let _ = self.pubsub.publish_local(&self.topic, self.our_id, buf);
        }
    }

    /// Request human approval. The prompt is emitted as a progress event;
    /// the human approver publishes an approval back to the same topic.
    pub async fn request_approval(&self, prompt: impl Into<String>) {
        self.progress(50, format!("APPROVAL_REQUIRED: {}", prompt.into())).await;
    }

    /// Publish to an arbitrary topic (not just the back-channel topic).
    /// Useful for broadcasting task lifecycle events to a broader audience
    /// while the RPC runs.
    pub async fn publish_topic(&self, topic: &str, event: Event) {
        let _ = self.pubsub.publish_local(topic, self.our_id, event.encode());
    }
}
```

The handler signature and builder method:

```rust
/// Handler that can emit back-channel events during a long RPC.
pub type BackchannelHandlerFn = Arc<
    dyn Fn(Request, Backchannel, CancellationToken) -> Pin<Box<dyn Future<Output = Result<Response, String>> + Send>>
        + Send + Sync,
>;

impl ServeBuilder {
    /// Register a back-channel handler for long-running RPCs.
    ///
    /// When a client sends an RPC request with the `EXT_BACKCHANNEL_TOPIC`
    /// extension, this handler is invoked with a `Backchannel` handle and a
    /// `CancellationToken`. The handler can emit progress events via the
    /// back-channel while the RPC runs; the final `Response` is returned on
    /// the RPC bi-stream as usual.
    ///
    /// If no back-channel handler is registered, requests with the extension
    /// fall back to the regular `handler()` (graceful degradation).
    pub fn backchannel_handler<F, Fut>(mut self, f: F) -> Self
    where
        F: Fn(Request, Backchannel, CancellationToken) -> Fut + Send + Sync + 'static,
        Fut: Future<Output = Result<Response, String>> + Send + 'static,
    {
        self.backchannel_handler = Some(Arc::new(move |req, bc, cancel| {
            Box::pin(f(req, bc, cancel))
        }));
        self
    }
}
```

### Part 5: Client API — `call_with_backchannel()` (P3, §5.4)

The client subscribes to the back-channel topic *before* issuing the RPC,
attaches the topic as an extension on the request, and receives a
`(ResponseFuture, ProgressStream)` pair.

```rust
// aafp-sdk/src/simple.rs

impl DiscoveryBuilder<'_> {
    /// Issue an RPC with a back-channel for progress events.
    ///
    /// Returns `(response_future, progress_stream)`. The progress stream
    /// yields `Event`s published by the server to the back-channel topic
    /// while the RPC runs. The stream closes when the RPC response arrives
    /// (or the connection drops).
    ///
    /// If the server does not support back-channeling, the progress stream
    /// simply yields no events and the response future resolves normally
    /// (graceful degradation).
    pub async fn call_with_backchannel(
        self,
        request: Request,
    ) -> Result<(ResponseFuture, ProgressStream), SdkError> {
        let server_id = self.resolve_server().await?;
        let req_id = generate_request_id();
        let bc_topic = backchannel_topic(&server_id, &req_id);

        // 1. Subscribe to the back-channel topic BEFORE sending the request.
        let progress = self.agent.subscribe(&bc_topic).await?;

        // 2. Build the RPC request frame with the back-channel extension.
        let rpc_bytes = encode_rpc_request(&req_id, &request)?;
        let frame = Frame::data(0, rpc_bytes);
        let frame = frame_with_backchannel(frame, &bc_topic);

        // 3. Open a bi-stream and send the request.
        let (mut send, recv) = self.agent.connection().open_bi().await?;
        send.write_all(&frame.encode()).await?;
        // Do NOT finish the send stream yet — keep it open for the response.

        // 4. Spawn a reader for the RPC response.
        let response_fut = ResponseFuture::new(recv, req_id.clone());

        // 5. Wrap the progress subscription in a ProgressStream that closes
        //    when the response arrives.
        let progress_stream = ProgressStream::new(progress, response_fut.clone());

        Ok((response_fut, progress_stream))
    }
}

/// Stream of progress events from a back-channel.
///
/// Wraps the PubSub subscription stream. Yields events until the RPC
/// response arrives or the subscription is dropped.
pub struct ProgressStream {
    inner: Pin<Box<dyn Stream<Item = Result<Event, SdkError>> + Send>>,
}

impl Stream for ProgressStream {
    type Item = Result<Event, SdkError>;
    fn poll_next(mut self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Option<Self::Item>> {
        self.inner.as_mut().poll_next(cx)
}
```

### Part 6: Progress Events Delivered via PubSub Subscription (P3, §5.6)

The back-channel does not use a new transport — it reuses the existing PubSub
publish path. The full sequence:

```
Client                                       Server
  |                                            |
  |--- subscribe(rpc.S.<id>.progress) -------->|  [PubSub subscribe, bi-stream]
  |                                            |
  |--- RPC_REQUEST(id, +bc_topic ext) -------->|  [bi-stream A]
  |                                            |  [Handler starts, gets Backchannel]
  |<-- publish(progress 10%) ------------------|  [PubSub publish, separate stream]
  |<-- publish(progress 50%) ------------------|
  |<-- publish(APPROVAL_REQUIRED) -------------|  [HITL prompt]
  |--- publish(approved) --------------------->|  [PubSub back to server]
  |                                            |  [Handler resumes]
  |<-- RPC_RESPONSE(id, final) ----------------|  [bi-stream A]
  |                                            |
```

Key points the implementation must enforce:
- The client subscribes to the back-channel topic **before** sending the RPC
  request. If the subscribe and request are reordered, early progress events
  are lost.
- Progress events are delivered as `aafp.pubsub.publish` RPC requests on
  **separate** bi-streams from the RPC response stream. The RPC bi-stream
  carries only the request and the final response.
- The client verifies `event.from()` equals the server's AgentId (§11.3
  spoofing mitigation). Events from other agents on the back-channel topic are
  ignored (or logged as suspicious).

```rust
/// Client-side filter: only accept progress events from the expected server.
fn filter_progress_events(
    stream: impl Stream<Item = Result<Event, SdkError>>,
    expected_server: AgentId,
) -> impl Stream<Item = Result<Event, SdkError>> {
    stream.filter(move |res| {
        let ok = match res {
            Ok(ev) => ev.from() == expected_server,
            Err(_) => true, // pass errors through
        };
        async move { ok }
    })
}
```

### Part 7: MQTT-Style Topic Hierarchies (P4, §6.1)

Topics are slash-separated (`.` is also accepted as a separator for
backwards compat with P1-P2 flat topics, but `/` is the canonical separator
for hierarchical topics). Hierarchical examples:

```
agents/region/us/capability/ocr       # OCR-capable agents in US region
agents/<agent_id>/status              # presence/status
agents/<agent_id>/inbox               # direct message (AutoGen-style)
llm/<model>/tokens                    # token stream
rpc/<server_id>/<req_id>/progress     # back-channel
tasks/<task_id>/events                # task lifecycle
tools/<tool_name>/events              # tool execution events
```

Add topic validation:

```rust
// aafp-messaging/src/topic.rs (new module)

/// Maximum topic length in bytes (§6.5).
pub const MAX_TOPIC_LENGTH: usize = 256;
/// Maximum topic depth (number of segments) (§6.5).
pub const MAX_TOPIC_DEPTH: usize = 16;

/// Validate a topic name for publishing.
/// Returns `Ok(())` if valid, or an error code (RFC-0005 extension).
pub fn validate_publish_topic(topic: &str) -> Result<(), u16> {
    if topic.is_empty() {
        return Err(9009); // PUBSUB_INVALID_TOPIC
    }
    if topic.len() > MAX_TOPIC_LENGTH {
        return Err(9007); // PUBSUB_TOPIC_TOO_LONG
    }
    let segs = split_topic(topic);
    if segs.len() > MAX_TOPIC_DEPTH {
        return Err(9009);
    }
    // Publish topics must not contain wildcards.
    if segs.iter().any(|s| *s == "+" || *s == "#") {
        return Err(9009);
    }
    Ok(())
}

/// Validate a subscription filter (allows wildcards).
pub fn validate_filter(filter: &str) -> Result<(), u16> {
    if filter.is_empty() {
        return Err(9009);
    }
    if filter.len() > MAX_TOPIC_LENGTH {
        return Err(9007);
    }
    let segs = split_topic(filter);
    if segs.len() > MAX_TOPIC_DEPTH {
        return Err(9009);
    }
    // `#` must be the last segment.
    for (i, s) in segs.iter().enumerate() {
        if *s == "#" && i != segs.len() - 1 {
            return Err(9009);
        }
    }
    Ok(())
}

/// Split a topic into segments. Accepts both `/` and `.` as separators
/// (normalizes `.` to `/` for hierarchical matching).
pub fn split_topic(topic: &str) -> Vec<&str> {
    topic.split(|c| c == '/' || c == '.').collect()
}
```

### Part 8: Wildcard Matching — `+` and `#` (P4, §6.3)

Borrowed from MQTT/RabbitMQ topic exchanges:

- `+` — **single-level wildcard**. Matches exactly one segment.
  `agents/+/status` matches `agents/A/status` and `agents/B/status` but not
  `agents/A/inbox` or `agents/A/sub/status`.
- `#` — **multi-level wildcard**. Must be the last segment. Matches zero or
  more remaining segments. `tasks/#` matches `tasks`, `tasks/123`, and
  `tasks/123/events`.

### Part 9: `TopicMatcher` Implementation (P4, §6.3, Appendix B)

Implement a `TopicMatcher` that stores both exact and wildcard subscriptions
and, on publish, finds all matching subscribers. Matching is segment-by-segment
(O(depth), negligible vs. floodsub forwarding cost).

```rust
// aafp-messaging/src/topic.rs

/// MQTT-style topic filter matching.
///
/// `+` matches exactly one level. `#` matches zero or more remaining levels
/// and must be the last segment of the filter.
pub fn topic_matches(filter: &str, topic: &str) -> bool {
    let f: Vec<&str> = split_topic(filter);
    let t: Vec<&str> = split_topic(topic);
    let mut fi = 0;
    for seg in &t {
        match f.get(fi) {
            Some(&"#") => return true,           // multi-level: rest matches
            Some(&"+") => fi += 1,               // single-level: match any
            Some(s) if *s == **seg => fi += 1,   // exact segment match
            _ => return false,
        }
    }
    // All topic segments consumed. Filter matches if fully consumed, or if
    // the only remaining filter segment is `#` (matches zero remaining).
    fi == f.len() || matches!(f.get(fi), Some(&"#"))
}

/// A matcher that indexes subscriptions by filter and resolves which
/// subscribers receive a given published topic.
///
/// Stores exact subscriptions in a `HashMap` for O(1) lookup, and wildcard
/// subscriptions in a `Vec` scanned on each publish (O(wildcards × depth)).
/// A trie index is a future optimization for large wildcard counts (§12.4).
pub struct TopicMatcher<T: Clone> {
    /// Exact topic -> subscribers.
    exact: HashMap<String, Vec<T>>,
    /// Wildcard filters -> subscribers.
    wildcard: Vec<(String, Vec<T>)>,
}

impl<T: Clone> TopicMatcher<T> {
    pub fn new() -> Self {
        Self { exact: HashMap::new(), wildcard: Vec::new() }
    }

    /// Register a subscriber for a filter. If the filter contains no
    /// wildcards, it goes in the exact map; otherwise in the wildcard vec.
    pub fn subscribe(&mut self, filter: &str, sub: T) {
        if filter.contains('+') || filter.contains('#') {
            if let Some(entry) = self.wildcard.iter_mut().find(|(f, _)| *f == filter) {
                entry.1.push(sub);
            } else {
                self.wildcard.push((filter.to_string(), vec![sub]));
            }
        } else {
            self.exact.entry(filter.to_string()).or_default().push(sub);
        }
    }

    /// Remove a subscriber from a filter. Returns true if a subscriber was
    /// removed.
    pub fn unsubscribe(&mut self, filter: &str, sub: &T) -> bool
    where T: PartialEq {
        let removed_from_exact = self.exact.get_mut(filter).is_some_and(|v| {
            let before = v.len();
            v.retain(|s| s != sub);
            v.len() < before
        });
        let removed_from_wild = self.wildcard.iter_mut().find(|(f, _)| *f == filter).is_some_and(|(_, v)| {
            let before = v.len();
            v.retain(|s| s != sub);
            v.len() < before
        });
        removed_from_exact || removed_from_wild
    }

    /// Find all subscribers whose filter matches the given published topic.
    /// Returns a Vec of cloned subscriber references.
    pub fn matches(&self, topic: &str) -> Vec<T> {
        let mut out = Vec::new();
        // Exact match: O(1).
        if let Some(subs) = self.exact.get(topic) {
            out.extend_from_slice(subs);
        }
        // Wildcard match: O(wildcards × depth).
        for (filter, subs) in &self.wildcard {
            if topic_matches(filter, topic) {
                out.extend_from_slice(subs);
            }
        }
        out
    }

    /// Number of distinct filters (exact + wildcard).
    pub fn filter_count(&self) -> usize {
        self.exact.len() + self.wildcard.len()
    }

    /// Total subscriber count across all filters.
    pub fn subscriber_count(&self) -> usize {
        self.exact.values().map(|v| v.len()).sum::<usize>()
            + self.wildcard.iter().map(|(_, v)| v.len()).sum::<usize>()
    }
}

impl<T: Clone> Default for TopicMatcher<T> {
    fn default() -> Self { Self::new() }
}
```

Wire `TopicMatcher` into `NetworkedPubSub`:

```rust
// In NetworkedPubSub, replace the flat HashMap<Topic, ...> with:
struct PubSubState {
    /// Exact + wildcard subscription matching.
    matcher: TopicMatcher<SubscriberId>,
    /// Local broadcast channels, keyed by filter (exact or wildcard).
    channels: HashMap<String, broadcast::Sender<TopicMessage>>,
}

// On publish, resolve all matching subscribers:
fn deliver(&self, topic: &str, msg: &TopicMessage) {
    for filter in self.state.matcher.matches(topic) {
        if let Some(tx) = self.state.channels.get(&filter) {
            let _ = tx.send(msg.clone()); // broadcast: laggy subscribers drop
        }
    }
}
```

### Part 10: Topic Naming Conventions and Reserved Prefixes (P4, §6.2)

Reserved top-level segments (registered in a future RFC amendment):

| Prefix    | Meaning                                       |
|-----------|-----------------------------------------------|
| `agents`  | Per-agent topics (status, inbox, capability)  |
| `rpc`     | Back-channel topics for in-flight RPCs        |
| `tasks`   | Task lifecycle events                         |
| `llm`     | LLM generation events (tokens, completions)   |
| `tools`   | Tool execution events                         |

Unprefixed topics are application-defined. The implementation should:

1. **Validate** reserved prefixes on subscribe/publish — `rpc.` topics require
   the 4-segment back-channel shape; `agents/` topics require a valid agent id
   in segment 2.
2. **Document** the conventions in doc-comments on the `Event` and `subscribe`
   API, but do not reject unprefixed topics (forward-compatible).
3. **Log a warning** (not an error) when a reserved prefix is used with an
   unexpected shape, to aid debugging without breaking apps.

```rust
/// Check if a topic uses a reserved prefix correctly.
/// Returns `Ok(())` if valid or unprefixed; `Err(warning_msg)` if a reserved
/// prefix is used incorrectly (non-fatal).
pub fn check_reserved_prefix(topic: &str) -> Result<(), String> {
    let segs = split_topic(topic);
    match segs.first() {
        Some(&"rpc") if !is_backchannel_topic(topic) => {
            Err(format!("rpc/ prefix should be rpc/<server>/<req_id>/progress, got: {topic}"))
        }
        Some(&"agents") if segs.len() < 2 => {
            Err(format!("agents/ prefix requires an agent id in segment 2, got: {topic}"))
        }
        _ => Ok(()),
    }
}
```

### Part 11: Integration with Streaming RPC (P4, §7.1)

A streaming RPC (server-streaming) is conceptually a PubSub subscription to a
topic with exactly one subscriber (the caller), where the topic is scoped to
the request. The back-channel design makes this explicit:
`rpc/<server>/<req_id>/progress` *is* a PubSub topic. The response stream
itself can be viewed as a private topic `rpc/<server>/<req_id>/response` with
a single consumer.

Unification benefits to realize in the implementation:
- **One frame-reading loop** serves both streaming RPC responses and PubSub
  events. Factor out a shared `read_frame_loop` utility.
- **Cancellation** (QUIC stream reset) and **backpressure** (QUIC flow control)
  apply uniformly. The `CancellationToken` from streaming RPC is reused for
  back-channel handlers.
- **The `Stream<Item = Result<Event/Response>>` ergonomics** are shared. Both
  `ProgressStream` and `ResponseStream` implement `Stream` and close on the
  same conditions (response arrival, connection drop, cancellation).

The implementations remain separate code paths for now (streaming RPC uses a
dedicated bi-stream; PubSub uses the propagation driver), but share utilities.
Do NOT merge the transport paths — only share the frame-reading and
cancellation logic.

```rust
/// Shared frame-reading loop for streaming RPC responses and PubSub events.
/// Reads frames from `recv`, decodes each, and sends to `tx`. Stops when
/// the stream ends, `cancel` fires, or `tx` is closed (consumer dropped).
async fn read_frame_loop(
    mut recv: RecvStream,
    tx: mpsc::Sender<Result<Frame, SdkError>>,
    cancel: CancellationToken,
) {
    loop {
        tokio::select! {
            _ = cancel.cancelled() => break,
            r = read_frame(&mut recv) => match r {
                Ok(Some(frame)) => {
                    if tx.send(Ok(frame)).await.is_err() { break; } // consumer gone
                }
                Ok(None) => break, // stream ended
                Err(e) => {
                    let _ = tx.send(Err(e.into())).await;
                    break;
                }
            }
        }
    }
}
```

## Constraints

1. **No wire protocol changes.** The back-channel uses existing RFC-0009
   publish frames on separate QUIC bi-streams. The only addition is an
   RFC-0006 extension (`EXT_BACKCHANNEL_TOPIC = 0x0010`) on the RPC_REQUEST
   frame, which is backward compatible by design (unknown extensions are
   ignored per RFC-0006 §3).

2. **Backward compatible.** Existing unary RPC code paths are untouched.
   Servers without back-channel support process the RPC normally (extension
   ignored). Clients without back-channel requests send no extension. Both
   directions degrade gracefully.

3. **No new transport.** Progress events are delivered via the existing
   PubSub publish path (floodsub propagation driver from P2). Do not invent a
   new QUIC stream type for back-channel events.

4. **Security: unguessable request ids.** The `req_id` in back-channel topics
   must be 64+ random bits. This prevents topic guessing attacks (§11.3).
   Clients should verify `event.from()` equals the server's AgentId.

5. **Wildcard matching is O(depth).** Use segment-by-segment comparison. A
   trie index is explicitly a *future* optimization (§12.4) — do not
   implement it now; the `Vec<(filter, subs)>` scan is sufficient for v1.

6. **`#` must be last in valid filters.** The validator rejects filters where
   `#` is not the final segment. `topic_matches()` itself is lenient (treats
   `#` as "match all remaining") but the subscription path validates first.

7. **Follow existing conventions.** Check `AGENTS.md`. Use `cargo fmt`,
   `cargo clippy`. Match the style of P2.8 (streaming RPC) code.
