# A-7 Completion Report: Normative Extension Processing Order

## Summary

**Item:** A-7 — Extension processing order (signature before semantics)
**Status:** DONE
**Verdict:** PASS

## Problem

Extensions could be processed before authentication, creating a
vulnerability window where extension-parsing bugs are exploitable via
unsigned messages. The RFC did not normatively specify the processing
order, leaving implementations to decide independently.

## Solution

Added a 20-phase normative frame processing pipeline to RFC-0002 §6.5.
The pipeline enforces that extension semantics execute only in Phase 18,
after all authentication and authorization phases (9-14) have succeeded.

## Security Invariant Verification

The core security property — "extension semantics MUST NOT execute
before authentication" — is verified by:

1. **Structural enforcement:** Extension callbacks are only invoked in
   Phase 18, which is after all authentication phases (9-14).

2. **Test verification:** The `extension_callbacks_invoked()` method
   returns false for all errors in Phases 1-17. All 81 Rust tests and
   58 Go tests verify this.

3. **Adversarial tests:** 25 Rust + 18 Go adversarial tests specifically
   attempt to bypass authentication using various attack vectors
   (critical extension injection, known extension bypass, etc.). All
   are rejected before Phase 18.

4. **Differential tests:** 20 cross-implementation test vectors verify
   that both Rust and Go produce identical results, including the
   callback count.
