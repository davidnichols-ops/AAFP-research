SECTION 1: CONTEXT
================================================================================

AAFP has never been tested with more than a few agents. There is no load
testing, no monitoring, no deployment documentation, and no operational
runbooks. This track makes AAFP production-ready.

Current state:
  - 1138+ tests, all functional (none test load/stress)
  - Performance benchmarks: 41.47µs round-trip, 776K msg/s (1 agent pair)
  - No metrics collection, no structured logging
  - No Dockerfile, no systemd service, no Kubernetes manifest
  - No operational documentation

### S1: Load Test Harness

WHAT TO BUILD:
  1. tests/load_test.rs:
     - LoadTestConfig: num_agents, messages_per_agent, message_size,
       duration, topology (mesh, star, ring, random)
     - LoadTestRunner: starts N agents, connects them, sends messages,
       collects metrics
     - LoadTestMetrics: throughput, latency p50/p90/p99, error rate,
       memory, CPU, connection count

  2. scripts/load-test.sh:
     - Starts N agents (separate processes or tokio tasks)
     - Runs load test
     - JSON output to test-results/performance/load-test-N.json

  3. Topologies:
     - Mesh: every agent → every other (limit to 10 connections each)
     - Star: all → central hub
     - Ring: each → neighbor
     - Random: each → K random peers

  4. Start with 10 agents, 100 messages each, 1KB, mesh topology

KEY FILES TO READ:
  implementations/rust/crates/aafp-sdk/src/agent.rs
    - Agent, AgentClient, AgentServer
  implementations/rust/crates/aafp-sdk/src/connection_pool.rs
    - ConnectionPool (I5) — used by load test
  implementations/rust/crates/aafp-benchmark/benches/mcp_transport.rs
    - Existing benchmark pattern for measuring throughput

### S2: 100-Agent Load Test

WHAT TO BUILD:
  1. Run load test with:
     - 100 agents
     - 1000 messages per agent
     - 1KB messages
     - 60 seconds duration
     - Topologies: star, ring, random (K=5)

  2. Measure:
     - Total messages sent/received
     - Throughput (msg/s aggregate)
     - Latency p50/p90/p99/p99.9
     - Error rate (lost messages, failed connections)
     - Memory per agent (<10MB target)
     - CPU per agent
     - File descriptors per agent

  3. Identify bottlenecks:
     - CPU-bound? (handshake/encryption)
     - Memory-bound? (buffer pool)
     - Network-bound? (QUIC streams)
     - Lock contention? (ArcSwap, Mutex)

  4. Write results to test-results/performance/load-test-100-agents.json

  5. If 100 agents is too many for one machine:
     - Start with 50, document what's needed for 100
     - Use multiple machines if available

### S3: Long-Running Stability Test

WHAT TO BUILD:
  1. Run one agent for 4-24 hours:
     - Accept connections from 10 clients
     - Each client sends 1 msg/s
     - Log metrics every 5 min: memory, CPU, FDs, connections

  2. Check for:
     - Memory growth (leak)
     - FD growth (connection leak)
     - CPU drift
     - Connection stability

  3. If leaks found:
     - Use cargo instruments (macOS) or heaptrack (Linux) for profiling
     - Fix the leak
     - Re-run stability test

  4. Write results to test-results/performance/stability-24h.json

### S4: Metrics and Observability

WHAT TO BUILD:
  1. AgentMetrics struct (in aafp-sdk/src/metrics.rs):
     - connections_active, connections_total
     - messages_sent, messages_received
     - bytes_sent, bytes_received
     - handshakes_completed, handshakes_failed
     - dht_records, relay_connections
     - uptime_seconds
     - Use AtomicU64 (lock-free)

  2. Agent::metrics() → AgentMetrics
  3. Agent::health_check() → HealthStatus (Healthy/Degraded/Unhealthy)

  4. Metrics endpoint:
     - aafp.metrics RPC method (returns CBOR)
     - Optional HTTP /metrics (Prometheus format, behind feature flag)

  5. Structured logging:
     - Use tracing crate (already available via tokio)
     - Log: connections, handshakes, errors, DHT ops
     - Configurable level: error/warn/info/debug/trace

  6. Tests:
     - Metrics are accurate after 100 messages
     - Health check returns Healthy after start
     - Health check returns Degraded if error rate > 10%

KEY FILES:

  implementations/rust/crates/aafp-sdk/src/lib.rs

### S6: Operational Runbook

WHAT TO BUILD:
  1. docs/OPERATIONS.md:
     - Key rotation (reference Track P5)
     - Rolling update (drain → update → restart)
     - Scaling (add agents)
     - Debugging slow agent (metrics → logs → CPU/mem → network)
     - Security incident (revoke → notify → rotate)
     - Relay management
     - DHT maintenance
     - Backup and recovery

  2. docs/TROUBLESHOOTING.md:
     - "Can't connect" → firewall, NAT, relay
     - "High latency" → network, congestion, DHT routing
     - "Memory growth" → leaks, connection accumulation
     - "Handshake fails" → key validity, clock skew, replay cache
     - "DHT lookup fails" → bootstrap, peers, TTL

### S7: Stress Testing

WHAT TO BUILD:
  tests/stress_test.rs with 5 stress scenarios:

  1. test_burst_traffic:
     - 100 agents × 100 messages simultaneously (10K burst)
     - Verify: no messages lost

  2. test_large_message:
     - Send 1MB, 10MB messages
     - Verify: frame handles large payloads, no OOM
     - Note: current max frame size is 1MB — document if needs change

  3. test_many_streams:
     - 100, 500, 1000 concurrent streams on one connection
     - Measure: memory per stream, quinn limits

  4. test_connection_churn:
     - 100 agents connect/disconnect every 1s
     - Measure: setup/teardown overhead, FD recovery

  5. test_dht_under_load:
     - 100 agents announce simultaneously
     - 100 agents lookup simultaneously
     - Measure: DHT throughput, latency

  Write results to test-results/performance/stress-tests.json
