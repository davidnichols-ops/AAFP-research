  2. Send PEX request to each seed

### R3: Record replication and republishing

Make records survive node failures.

- Implement record replication:
  - When agent announces, record stored on k closest nodes (default k=5)
  - `announce()` forwards record to k closest peers
  - Each receiving peer stores record locally

- Implement republishing:
  - Agents re-announce before TTL expires
  - Default TTL: 1 hour, republish interval: 30 minutes
  - Background task: republish all own records every 30 minutes

- Implement record refresh:
  - When node joins, request records from peers
  - `aafp.discovery.refresh` RPC: "send me all records you have"
  - New node populates store from refresh responses

- Implement active expiration:
  - Every 5 minutes, evict expired records
  - Log when records expire

- **VERIFY:** Agent A announces to node N1. N1 replicates to N2-N5. N1 goes down. Agent B can still look up A's record from N2-N5.

Key files:
- `crates/aafp-discovery/src/discovery_v1.rs` — add replication logic
- `crates/aafp-discovery/src/rpc_handler.rs` — add refresh RPC

### R4: Churn handling

Handle agents going offline and coming back.

- Implement peer liveness tracking:
  - Ping each peer every 60 seconds (use PING/PONG from E1)
  - 3 consecutive missed pings → mark as dead
  - Remove dead peers from routing table
  - Trigger bucket refresh for dead peer's bucket

- Implement re-announcement on rejoin:
  - When agent comes back online, re-announces to k closest peers
  - Peers update timestamp, store if new

- Implement graceful departure:
  - `Agent::leave()` — announce departure to k closest peers
  - Peers remove agent's record immediately
  - `aafp.discovery.depart` RPC method

- Implement routing table repair:
  - When bucket has fewer than k entries, query known peers for more
  - Background task: every 15 minutes, check all buckets, repair underfilled

- **VERIFY:** Agent A goes offline (3 missed pings), peers detect within 3 minutes, remove A's records. A comes back, re-announces, records visible within 1 minute.

Key files:

Optimize DHT routing for production scale.

- Implement parallel lookups:

### R6: Network partition handling

Handle network splits and reconciliation.

- Implement partition detection:
  - Single peer unreachable = peer failure (R4)
  - Many peers unreachable simultaneously = partition
  - Log partition events

- Implement record reconciliation:
  - When partition heals, peers exchange records via refresh (R3)
  - Conflicting records (same agent_id, different public_key): latest timestamp wins, log warning
  - Missing records: request from peers

- Implement split-brain prevention:
  - During partition, both sides accept announces independently
  - On heal: merge records (union, latest timestamp wins)
  - No rollback — both sides' records are valid

- **VERIFY:** 10 nodes split into 2 groups of 5. Each group accepts announces. Partition heals. All 10 nodes have all records.

## Research Insights

- **Kademlia DHT churn:** Kademlia is robust to churn but requires active bucket refresh. Without refresh, stale entries accumulate and lookups fail. R4's bucket repair is critical.
- **GossipSub v1.1 peer scoring:** If AAFP later adds gossip-based pub/sub, peer scoring (mesh delivery, app-specific, IP colocation, behavioral) becomes important. For now, the DHT's k-bucket approach is sufficient.
- **DHT keyspace:** AAFP's DHT is keyed by capability string (hashed to 256-bit), not by agent_id. There may be only 10-100 distinct capabilities, so the keyspace is small. Consider a simpler gossip-based approach if Kademlia is overkill. The risk is noted in the track plan.
