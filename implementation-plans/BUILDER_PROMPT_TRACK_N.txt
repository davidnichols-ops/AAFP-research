## Step Details

### N1: Relay Data Forwarding (THE CRITICAL STEP)

This is the most important step. The relay must forward real QUIC streams.

**What to build:**

1. **`RelayV1Service::forward_data()` async method:**
   - Takes two `(QuicSendStream, QuicRecvStream)` pairs (caller and target)
   - Spawns two tokio tasks: caller→target and target→caller
   - Uses `tokio::io::copy()` for efficient byte copying
   - Tracks `bytes_forwarded` in the `RelayedConnection`
   - Closes both sides when either closes

2. **`handle_relayed_connection()` async function:**
   - Called when a caller requests a relayed connection to target B
   - The relay has its own `QuicTransport` for dialing targets
   - Dials target B using the relay's transport
   - Opens a bi-stream to target B
   - The caller's stream is already open (from the RPC request)
   - Calls `forward_data()` to copy bytes bidirectionally
   - Returns when the connection closes

3. **Modify `RelayV1RpcHandler` to support async data forwarding:**
   - Current handler is synchronous (returns CBOR Value)
   - Data forwarding is async and long-lived (stream, not request/response)
   - Approach: The `handle_connect()` RPC returns a `ConnectResult` with
     `connection_id`. Then the caller opens a bi-stream on the same QUIC
     connection with a header indicating `connection_id`. The relay accepts
     bi-streams, reads the connection_id, and starts forwarding.
   - Alternative approach: The relay opens a bi-stream back to the caller
     after `handle_connect()` succeeds. Simpler but requires the relay to
     hold the caller's `QuicConnection`.

4. **Connection lifecycle management:**
   - Track active relayed connections in `RelayV1Service`
   - Enforce `max_connections` (default 50)
   - Clean up when either side closes
   - Track bytes forwarded for metrics

**Key design decision:** How does the caller's data reach the relay?

The relay is a QUIC server. The caller connects to the relay via QUIC. On that QUIC connection:
- Stream 0 (or a control stream): Used for RPC (reserve, connect, etc.)
- After `connect` RPC succeeds, the caller opens a NEW bi-stream for data
- The relay accepts this bi-stream, reads the connection_id header, and
  starts forwarding to the target

Wire format for data stream header:
```
[1 byte: 0xFF (magic)] [8 bytes: connection_id (big-endian)]
```
Then raw bytes follow, which the relay copies to the target's stream.

**VERIFY:** A test where agent A → relay → agent B can send and receive data through the relay (localhost). The test should:
1. Start relay node (QuicTransport + RelayV1Service)
2. Start agent B (QuicTransport + RelayV1RpcHandler for incoming relay connections)
3. Agent B reserves on the relay
4. Agent A connects to the relay, requests relayed connection to B
5. Agent A sends a message through the relay
6. Agent B receives the message
7. Agent B sends a reply
8. Agent A receives the reply
9. Close the connection

### N3: DCuTR Hole Punching

**What to build:**

1. **`Dcutr::initiate_upgrade()` async method:**
   - Both peers exchange their observed external addresses (via relay)
   - Both peers simultaneously dial each other's observed addresses
   - If both dials succeed, upgrade to direct connection
   - If either fails, keep the relayed connection

2. **Address exchange protocol (via relay):**
   - `aafp.dcutr.observe` — Ask peer to report my observed address
   - `aafp.dcutr.connect` — Initiate simultaneous open with exchanged addresses

3. **Simultaneous open:**
   - Use `QuicTransport::dial()` with the peer's observed address
   - Both sides dial at the same time (coordinated via relay RPC)
   - Quinn handles the rest — if both dials reach the NAT, holes are punched
   - Use quinn's `Connection::migrate()` if needed for path validation

4. **NAT type detection:**
   - If AutoNAT reports both peers as cone NAT → attempt DCuTR
   - If either is symmetric NAT → skip DCuTR, keep relay
   - Document that symmetric NAT requires relay (unfixable without port prediction)

5. **Research insight (from WORLD_SCALE_RESEARCH.md):**
   - DCUtR success rate: 70% ± 7.1% for hole punching (from 4.4M attempts in IPFS)
   - TCP and QUIC have comparable success rates (~70%)
   - 97.6% of successful connections established on first attempt
   - AAFP should target 70% hole-punch + 20% relay fallback = 90% success

## Research Insights (apply during implementation)

From `implementation-plans/WORLD_SCALE_RESEARCH.md`:

1. **DCUtR success rate:** 70% ± 7.1% for hole punching (4.4M attempts in IPFS).
   TCP and QUIC have comparable success rates. 97.6% of successes on first attempt.
   Target: 70% hole-punch + 20% relay fallback = 90% success.

2. **QUIC connection migration failure:** 76% failure rate on hard handoffs (Wi-Fi→Wi-Fi).
   For N3 (DCuTR): use active migration initialization, not passive.
   For N7: expect failures and document fallback (reconnect with session resumption).

3. **Relay capacity:** WebRTC TURN servers handle 50K-100K concurrent flows.
   coturn: 10K+ sessions on 4 vCPU/8GB RAM.
   AAFP's default max_connections=50 is conservative — can increase for production.

4. **NAT types in the wild:**
   - 15-20% of consumer connections need relay (NAT traversal fails)
   - 40-60% of enterprise connections need relay
   - 85% in strict corporate environments
   - Symmetric NAT: ~20% of NATs (DCuTR cannot work)
